# Phase 4.1: Model Switching - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a `/model` command that lets users switch the active AI model globally via the OpenCode config API, and list available models grouped by provider. `/clear` was dropped from this phase — users use `/new` instead.

Requirements in scope: FILE-02 (`/model <name>`)

</domain>

<decisions>
## Implementation Decisions

### `/model <name>` — Switch Model
- **D-01:** `/model <name>` calls `PATCH /config` on OpenCode — model change is **global** (applies to all sessions, not per-session). This matches OpenCode's actual config API shape.
- **D-02:** On success: `✅ Model switched to <providerID>/<modelID>` (terse single-line, consistent with `/status` format)
- **D-03:** On failure / unknown model: `❌ Unknown model "<name>". Run /model to see available models.`

### `/model` (no argument) — Show Current + List
- **D-04:** `/model` with no argument shows the current active model AND the full list of available models from `GET /config/providers`, grouped by provider.
- **D-05:** Available models are formatted grouped by provider — e.g.:
  ```
  Model: anthropic/claude-sonnet-4

  Available models:
  Anthropic: claude-sonnet-4, claude-opus-4
  OpenAI: gpt-4o, gpt-4o-mini
  ```
- **D-06:** Current model is shown on the first line in the same `providerID/modelID` format used by `/status`.

### `/clear` — Dropped
- **D-07:** `/clear` is NOT implemented in this phase. Users who want a fresh context use `/new <name>` to create a new session. ROADMAP updated to remove `/clear` from phase 4.1 scope.

### Claude's Discretion
- Exact `PATCH /config` request body shape — researcher to confirm from live OpenCode API (likely `{ model: { providerID, modelID } }` or similar)
- How `GET /config/providers` response is shaped — researcher to confirm and map to display format
- File/module location for the new handler (follow existing pattern: `src/bot/handlers/cmd-model.ts`)
- Whether to add `/model` to the BotFather command menu via `setMyCommands` (probably yes — follow Phase 4 pattern)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/bot/handlers/cmd-status.ts` — `fetchActiveModel(baseUrl, sessionId)` already fetches model info from session messages; the new `/model` handler should use a different approach (config API) but can reference this for patterns
- `src/opencode/session.ts` — existing OpenCode HTTP client; new `patchConfig()` and `getConfigProviders()` functions follow the same fetch pattern
- `src/session/registry.ts` — `SessionRegistry` for looking up active session ID per chat (needed to show current model in `/model` no-arg output)
- `src/bot/index.ts` — `createBot(manager, registry, url)` factory; new `bot.command("model", ...)` registration goes here
- `src/logger.ts` — shared pino logger

### Established Patterns
- ESM project: all imports use `.js` extensions even for TypeScript source files
- Handler factory: `makeModelHandler(registry, url)` returning async grammY handler — testable via DI
- Error messages: `❌` prefix for errors, `✅` for success
- `ctx.match` for command argument extraction (e.g., `ctx.match` gives the text after `/model`)

### Integration Points
- `src/bot/index.ts`: add `bot.command("model", makeModelHandler(registry, openCodeUrl))` registration
- `src/opencode/session.ts` (or new `src/opencode/config.ts`): add `patchConfig(baseUrl, model)` and `getConfigProviders(baseUrl)` functions
- BotFather menu: add `/model` to `setMyCommands` call in `main.ts` (follow Phase 4 pattern)

</code_context>

<specifics>
## Specific Ideas

- `/model` no-arg output: current model on first line, then blank line, then grouped provider list — mirrors how `/sessions` shows state then a usage hint
- Model switching is global — explicitly document this in the bot's reply so users aren't confused when it affects other sessions

</specifics>
