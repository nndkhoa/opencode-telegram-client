# Phase 04.2: Fix unknown model at beginning; propose new way to change model - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

1. **Fix “Model: unknown” on `/status`** for new or empty sessions: derive display model from **global OpenCode config** when session messages do not yet contain an assistant model, instead of showing `unknown` only because there are no messages.

2. **Unify “current model” resolution** between `/status` and `/model` so both commands agree.

3. **New way to switch models:** keep `PATCH /config` as today, but extend `/model` so users can pick from a **numbered list** — e.g. `/model 2` selects the model labeled **2** in the latest `/model` listing.

4. **Copy:** When the model still cannot be resolved, keep `/status` readable with a **short hint** to use `/model` (exact wording left to implementation).

Out of scope: inline keyboards (deferred unless pulled into a later phase), per-session model override, changing OpenCode server behavior.

</domain>

<decisions>
## Implementation Decisions

### `/status` model resolution (empty / new session)
- **D-01:** Resolve display model in this **order**: (1) **`GET /config`** via `extractConfiguredModel` (same semantics as `/model` “current model”); (2) if still unset, **session messages** — last assistant message with `modelID` / `providerID` (existing `fetchActiveModel` logic); (3) if still unset, treat as **not set** for display (see D-05).
- **D-02:** When OpenCode is **unreachable** for `/status`, keep existing degraded behavior (`Model: unknown` / `State: unknown` as today); do not block the line on partial failures beyond what Phase 4 already defined.

### Single source of truth
- **D-03:** Introduce a **shared helper** (e.g. `resolveDisplayModel(baseUrl, sessionId)` or split internal steps) used by **`cmd-status.ts`** and any **`/model`**-related display logic so “current model” strings do not diverge.
- **D-04:** Unit tests cover: config-only, message-only, config+message agree, empty session + config set (the main bug), unreachable OpenCode.

### Numbered `/model` list and `/model <n>`
- **D-05:** When `/model` is run **with no argument**, the **available models** section assigns a **stable index** to every selectable model for PATCH: **1-based** integer labels (`1`, `2`, …) in a **deterministic flat order** that matches the order used to resolve `/model <n>` (see D-06).
- **D-06:** Build the flat list in **deterministic order**: e.g. same provider sort as today (`formatCatalogBlocks`), then models within each provider in a stable key order — document the exact rule in code comments so tests can lock it.
- **D-07:** **Argument disambiguation:** If the trimmed argument matches **`/^\d+$/`** (decimal digits only), treat it as a **catalog index** (1-based): resolve to the corresponding `provider/model` ref from the same flat list used for numbering, then `PATCH /config` with that ref. If the index is out of range, reply with a clear error (e.g. suggest running `/model` for the current list).
- **D-08:** If the argument is **not** all digits (e.g. contains `/` or letters), keep **existing** behavior: pass the string to `patchConfig` as the **full model id** (case-sensitive per Phase 04.1).
- **D-09:** Success line after switch by number: same style as today, e.g. `✅ Model switched to provider/model (global — affects all sessions).` — show the **resolved** ref, not only the number.

### Copy / hints
- **D-10:** When the display model is still **not set** after resolution, `/status` should include a **short hint** on the same line or as minimal extra text, e.g. pointing to `/model` (exact phrase: Claude’s discretion; keep single-line spirit of Phase 4 where possible).

### Claude's Discretion
- Exact helper module path and function names; whether `fetchActiveModel` stays private or is folded into the helper.
- Telegram HTML formatting for numbered bullets in `/model` output (must stay within parse limits and existing chunking).
- Exact error strings for out-of-range index vs unknown string model.

</decisions>
