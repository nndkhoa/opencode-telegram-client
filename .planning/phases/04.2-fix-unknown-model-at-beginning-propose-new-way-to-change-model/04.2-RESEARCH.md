# Phase 04.2: Fix unknown model at beginning; propose new way to change model — Research

**Researched:** 2026-03-29  
**Domain:** OpenCode config + session messages + Telegram `/status` / `/model` UX  
**Confidence:** HIGH (grounded in repo code + locked CONTEXT decisions)

## Summary

Today `/status` uses `fetchActiveModel` only, which reads the last assistant message from `GET /session/:id/message`. New or empty sessions have no assistant messages, so the line shows `Model: unknown` even when `GET /config` already exposes the effective global model via `extractConfiguredModel` (same semantics `/model` uses for “current model”). Phase 04.2 fixes this by resolving display model in **config-first, then messages** order (D-01), and centralizing that logic in a **shared helper** (D-03) so `/status` and `/model` never diverge.

The second pillar is a **deterministic flat catalog** of selectable `provider/model` refs, numbered `1..n` in listing output and in **`/model <n>`** when the trimmed argument matches `/^\d+$/` (D-05–D-08). Non-numeric arguments continue to pass through to `patchConfig` as the full model id (case-sensitive per 04.1).

**Primary recommendation:** Add `src/opencode/` (or `src/bot/`) modules exporting (1) `resolveDisplayModel` / internal steps with **per-step `try/catch`** so a failing `GET /config` does not force the entire `/status` reply into the unreachable/degraded path when `checkHealth` succeeded; (2) `buildFlatSelectableModelRefs(payload)` shared by numbered listing and index resolution; (3) extend `makeCmdModelHandler` to branch on `/^\d+$/` vs string. Lock ordering rules in code comments and mirror them in unit tests.

## Architecture Patterns

### Recommended module layout

```
src/opencode/
  config.ts              # existing — extractConfiguredModel, getConfig, patchConfig, …
  display-model.ts       # NEW (recommended name): resolveDisplayModel, fetchModelFromMessages, …
  or model-catalog.ts    # NEW: buildFlatSelectableModelRefs + shared sort rules

src/bot/handlers/
  cmd-status.ts          # call shared resolver; hint when unset (D-10)
  cmd-model.ts           # use same resolver for “Current model”; flat list + /model <n>
```

### Pattern: `resolveDisplayModel` (config → messages → unset)

**What:** Async function that returns a **display result** (not only a string), e.g. `{ kind: "resolved", ref: string } | { kind: "unset" }`, so `/status` can append a hint only for `unset` without overloading `unknown`.

**When to use:** Single place for D-01 ordering.

**Implementation notes:**

1. **`sessionId` absent:** Skip message fetch; only `getConfig` path (registry returns `undefined` for no session — see `cmd-status.test.ts`).
2. **`getConfig` throws:** Log (like `cmd-model` no-arg path). **Do not** let this throw propagate to `makeCmdStatusHandler`’s outer `catch`, or you would incorrectly show full unreachable degradation (D-02 vs partial API failure). Fall through to messages, then `unset`.
3. **`GET /session/.../message` fails:** Same as today — treat as missing model for that leg; final result may be `unset` or config-only.

### Pattern: deterministic `buildFlatSelectableModelRefs(payload)`

**What:** One ordered `string[]` of full refs (`provider/model`) used for:

- Assigning indices `1..n` in the no-arg `/model` HTML.
- Resolving `/model <n>` to the same ref before `patchConfig`.

**Order (align with D-06 and existing UX):**

1. Sort providers: `(a.name || a.id).localeCompare(b.name || b.id, undefined, { sensitivity: "base" })` — same as `formatCatalogBlocks` today.
2. Within each provider, sort model keys: `Object.keys(models).sort()` (locale-independent ASCII sort is stable; document choice in comment).
3. For each key `id`, full ref: `id.includes("/") ? id : \`${provider.id}/${id}\`` — same rule as `formatCatalogBlocks`.

**Anti-pattern:** Duplicating provider sort in one place and a different order in another — numbering and `/model <n>` will disagree.

### Pattern: digit-only argument disambiguation (D-07 vs D-08)

```typescript
const trimmed = arg.trim();
if (/^\d+$/.test(trimmed)) {
  const index = Number(trimmed); // "01" → 1 — 1-based
  // validate 1 <= index <= flat.length
} else {
  await patchConfig(baseUrl, trimmed);
}
```

**Edge case:** A model id that is **only digits** (hypothetical) would be treated as an index, not a string id — accepted tradeoff per D-07 (explicit user decision).

### Anti-patterns to avoid

- **Calling `fetchActiveModel` alone for `/status`** after this phase — bypasses config-first order (D-01).
- **Sorting `Object.keys(models)` differently** between HTML bullets and flat index list — breaks D-05/D-06.
- **Letting `getConfig` throw collapse `/status` into unreachable** when `checkHealth` succeeded — violates expected degraded semantics (fix with inner try/catch).

## Common Pitfalls

### Pitfall 1: `/status` outer try/catch and `getConfig`

**What goes wrong:** `checkHealth` succeeds; `getConfig` throws → entire handler hits `catch` → `OpenCode: ❌ unreachable` while server is up.

**Why:** Single try wrapping all async calls.

**How to avoid:** Isolate `getConfig` + `extractConfiguredModel` in inner try/catch inside `resolveDisplayModel` (or equivalent).

**Warning signs:** Tests for “healthy + config fetch failure” would show wrong health line.

### Pitfall 2: `formatCatalogBlocks` model order

**What goes wrong:** Today `Object.keys(models)` is not explicitly sorted; insertion order may vary if OpenCode changes object construction.

**Why:** D-06 requires **stable** key order.

**How to avoid:** Sort keys explicitly in both `formatCatalogBlocks` and `buildFlatSelectableModelRefs`.

### Pitfall 3: 0-based vs 1-based indexing

**What goes wrong:** Off-by-one or `/model 0` mapping to last item.

**Why:** D-05 specifies 1-based labels.

**How to avoid:** `if (n < 1 || n > flat.length)` → error; use `flat[n - 1]` after validation.

### Pitfall 4: Multi-message `/model` reply and numbering

**What goes wrong:** User sees continuation `… 2/3` without indices on later chunks — confusion about which number maps to which entry.

**Why:** `packTelegramHtmlSections` splits oversized catalogs.

**How to avoid:** Prefer numbering in **every** chunk’s “Available models” section, or document that users should run `/model` on a single screen when list is huge; ensure chunk boundaries do not split mid-numbered block if possible (optional polish — discretion).

## Code Examples

### Existing message-based model (message leg of D-01)

```7:21:src/bot/handlers/cmd-status.ts
async function fetchActiveModel(baseUrl: string, sessionId: string): Promise<string> {
  try {
    const res = await fetch(new URL(`/session/${sessionId}/message?limit=10`, baseUrl).toString());
    if (!res.ok) return "unknown";
    const msgs = (await res.json()) as Array<{
      info: { role: string; modelID?: string; providerID?: string };
    }>;
    // Model info is on assistant messages (info.modelID + info.providerID)
    const assistant = msgs.find(m => m.info.role === "assistant" && m.info.modelID);
    if (!assistant?.info.modelID) return "unknown";
    const { providerID, modelID } = assistant.info;
    return providerID ? `${providerID}/${modelID}` : modelID!;
  } catch {
    return "unknown";
  }
}
```

*Note:* For D-01, the merged resolver should return a structured result; the message leg can reuse this logic but return `undefined` instead of `"unknown"` when missing, so the final line can distinguish “unset” vs “OpenCode unreachable” (D-02).

### Config extraction (config leg of D-01)

```62:81:src/opencode/config.ts
export function extractConfiguredModel(cfg: unknown): string | undefined {
  if (cfg === null || typeof cfg !== "object") return undefined;
  const o = cfg as Record<string, unknown>;
  const top = o.model;
  if (typeof top === "string" && top.trim()) return top.trim();

  const agent = o.agent;
  if (agent === null || typeof agent !== "object" || Array.isArray(agent)) return undefined;
  const a = agent as Record<string, unknown>;
  const priority = ["build", "plan", "title", "summary", "compaction", "general", "explore"] as const;
  for (const key of priority) {
    const m = readAgentModelBlock(a[key]);
    if (m) return m;
  }
  for (const key of Object.keys(a).sort()) {
    if ((priority as readonly string[]).includes(key)) continue;
    const m = readAgentModelBlock(a[key]);
    if (m) return m;
  }
  return undefined;
}
```
